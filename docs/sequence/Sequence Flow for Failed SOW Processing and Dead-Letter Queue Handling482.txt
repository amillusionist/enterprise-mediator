# 1 Overview

## 1.1 Diagram Id

SEQ-EH-001

## 1.2 Name

Failed SOW Processing and Dead-Letter Queue Handling

## 1.3 Description

When the SOW Processing Worker fails to process a message after all configured retries (e.g., due to a malformed document or a persistent external API failure), the message is automatically moved to a Dead-Letter Queue (DLQ). An automated alert is triggered for manual intervention by the operations team.

## 1.4 Type

🔹 ErrorHandling

## 1.5 Purpose

To create a resilient asynchronous processing system that can handle unrecoverable errors without losing data, preventing poison messages from halting the entire workflow, and notifying operators of failures that require investigation.

## 1.6 Complexity

Medium

## 1.7 Priority

🔴 High

## 1.8 Frequency

OnDemand

## 1.9 Participants

- REPO-SVC-AIWORKER
- REPO-MQ-RABBITMQ
- REPO-INFRA-AWS

## 1.10 Key Interactions

- Worker fails to process a message from the main SOW processing queue due to a non-transient error.
- The message broker's retry mechanism re-delivers the message a configured number of times.
- After the final failed attempt, the message broker automatically moves the 'poison' message to the configured DLQ.
- A CloudWatch Alarm is configured to monitor the 'ApproximateNumberOfMessagesVisible' metric of the DLQ.
- When the metric is > 0 for a sustained period, the alarm transitions to an ALARM state.
- The alarm triggers a notification (e.g., to an SNS topic) which alerts an operator.
- An operator inspects the message content, reviews worker logs using the correlation ID, and decides on a resolution (e.g., fix a bug and re-queue, or discard).

## 1.11 Triggers

- A message in the SOW processing queue fails all configured retry attempts.

## 1.12 Outcomes

- The failing message is isolated in a DLQ, preventing it from blocking subsequent messages.
- An automated alert is sent to the operations team for investigation.
- No data (the SOW processing request) is lost, ensuring traceability.

## 1.13 Business Rules

- A message must only be dead-lettered after a defined number of retries has been exhausted.
- Upon permanent failure, the SOW's status should be updated to 'Failed' in the database with an error summary.

## 1.14 Error Scenarios

- This entire sequence describes the handling of an error scenario.

## 1.15 Integration Points

- Message Queue (RabbitMQ) DLQ feature.
- Monitoring and Alerting System (e.g., AWS CloudWatch).

#### 2.6.1.1.0.0 Content
