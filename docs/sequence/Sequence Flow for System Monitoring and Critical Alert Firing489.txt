# 1 Overview

## 1.1 Diagram Id

SEQ-OF-001

## 1.2 Name

System Monitoring and Critical Alert Firing

## 1.3 Description

A monitoring tool (e.g., Prometheus, CloudWatch) continuously scrapes system metrics. When a metric breaches a predefined threshold (e.g., API p99 latency > 1 second), an alert is fired and routed to the on-call team via a configured notification channel.

## 1.4 Type

🔹 OperationalFlow

## 1.5 Purpose

To ensure proactive issue detection and response, fulfilling the monitoring and alerting requirements of REQ-REP-002 and supporting the uptime NFR (REQ-NFR-004).

## 1.6 Complexity

Medium

## 1.7 Priority

🔴 High

## 1.8 Frequency

Continuously

## 1.9 Participants

- REPO-API-MONOLITH
- REPO-INFRA-PROMETHEUS
- REPO-INFRA-ALERTMANAGER

## 1.10 Key Interactions

- The API exposes a '/metrics' endpoint scraped by Prometheus.
- System load increases, causing API latency to rise.
- Prometheus records the high latency metric values.
- An alert rule in Prometheus (e.g., 'p99 latency > 1s for 5m') evaluates to true.
- Prometheus sends an alert to Alertmanager.
- Alertmanager groups, deduplicates, and routes the alert to a configured receiver, such as PagerDuty or Slack.
- The on-call engineer is notified.

## 1.11 Triggers

- A configured alert rule's condition is met.

## 1.12 Outcomes

- Operations team is notified of a potential or active system issue.
- Mean Time To Detection (MTTD) is minimized.

## 1.13 Business Rules

- Alert thresholds must align with the NFRs defined in the SRS (e.g., REQ-NFR-001).

## 1.14 Error Scenarios

- A 'flapping' alert fires and resolves repeatedly, causing noise.
- A misconfigured alert rule fails to fire during a real incident.

## 1.15 Integration Points

- Prometheus for metrics collection.
- Alertmanager for alert routing.

# 2.0 Details

## 2.1 Diagram Id

SEQ-OF-001

## 2.2 Name

Operational Flow: Critical API Latency Alert Firing

## 2.3 Description

This sequence details the end-to-end operational flow for detecting and alerting on critical performance degradation of the backend API, in alignment with SRE principles. The process begins with increased system load causing a violation of the API latency SLO (Service Level Objective). Prometheus, via its OpenTelemetry metric scraper, detects this breach, evaluates a pre-configured alerting rule, and fires an alert to Alertmanager. Alertmanager then processes, groups, and routes the alert to the designated external notification channel, ensuring the on-call team is immediately notified. This flow is a direct implementation of requirements REQ-REP-002 and REQ-NFR-004.

## 2.6.0.0 Notes

### 2.6.1.0 Content

#### 2.6.1.1 Content

The 'FOR: 5m' clause in the Prometheus alert rule is critical to prevent 'flapping' alerts from transient spikes. The alert only fires if the condition is continuously met for the entire duration.

#### 2.6.1.2 Position

Right

#### 2.6.1.3 Participant Id

REPO-INFRA-PROMETHEUS

#### 2.6.1.4 Sequence Number

4

### 2.6.2.0 Content

#### 2.6.2.1 Content

Alertmanager's grouping logic is key to reducing alert fatigue. For instance, if 10 API pods are slow, it sends one grouped notification, not 10 individual ones.

#### 2.6.2.2 Position

Right

#### 2.6.2.3 Participant Id

REPO-INFRA-ALERTMANAGER

#### 2.6.2.4 Sequence Number

6

### 2.6.3.0 Content

#### 2.6.3.1 Content

The `/metrics` endpoint is a pull-based model. It must be designed to be highly performant and lightweight to not impact application performance during scrapes.

Top

#### 2.6.3.3 Participant Id
