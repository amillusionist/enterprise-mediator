# 1 Overview

## 1.1 Diagram Id

SEQ-DF-001

## 1.2 Name

Scheduled Dashboard Metrics Calculation

## 1.3 Description

A scheduled background job runs periodically to aggregate data from various transactional tables (Projects, Invoices, Proposals, etc.) and populates a denormalized 'DashboardMetrics' table. This table is optimized for fast reads and powers the main admin dashboard.

## 1.4 Type

🔹 DataFlow

## 1.5 Purpose

To ensure the main administrative dashboard (REQ-FUN-005) loads quickly and meets strict performance NFRs (REQ-NFR-001) by pre-calculating and denormalizing key business intelligence metrics, avoiding complex, slow queries on user request.

## 1.6 Complexity

Medium

## 1.7 Priority

🟡 Medium

## 1.8 Frequency

Hourly

## 1.9 Participants

- REPO-INFRA-AWS
- REPO-API-MONOLITH
- REPO-DB-POSTGRES

## 1.10 Key Interactions

- A scheduler (e.g., Kubernetes CronJob or AWS EventBridge) triggers the job on a defined schedule (e.g., '0 * * * *').
- The job, a containerized service or function, is initiated.
- The service executes several optimized, read-only SQL queries against the transactional tables within a single transaction.
- The queries aggregate data such as COUNT of projects by status, SUM of revenue, number of pending SOWs, etc.
- The service writes the aggregated results to a dedicated 'DashboardMetrics' table, typically by truncating and inserting new data.
- The job logs its success or failure and the number of records processed, then completes.

## 1.11 Triggers

- A recurring schedule, such as a cron expression.

## 1.12 Outcomes

- The 'DashboardMetrics' table contains fresh, aggregated data.
- The main admin dashboard can load its widgets extremely quickly by querying this single, simple table.

## 1.13 Business Rules

- The job must be idempotent and transactional; a failure should not leave the metrics table in an inconsistent state.
- The job must be designed to not place excessive load on the primary database during its run.

## 1.14 Error Scenarios

- The aggregation queries fail due to a database lock or other issue, causing the job to fail and log an error.
- The job takes longer to run than its scheduled interval, potentially causing overlapping runs if not handled correctly.

## 1.15 Integration Points

- A cloud-native scheduling service (e.g., Kubernetes CronJob, AWS EventBridge).

## 2.2 Name

Implementation: Scheduled Dashboard Metrics Calculation

## 2.3 Description

A detailed technical sequence for a Kubernetes-scheduled background job that performs an ETL process. The job reads from multiple transactional tables within a PostgreSQL database, aggregates key business metrics, and populates a denormalized read-model table ('DashboardMetrics'). This sequence follows the CQRS pattern to optimize read performance for the main admin dashboard, ensuring adherence to strict performance NFRs (REQ-NFR-001).
