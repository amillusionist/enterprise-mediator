# 1 Overview

## 1.1 Diagram Id

SEQ-BP-001

## 1.2 Name

Admin Creates New Project from SOW

## 1.3 Description

An authenticated Admin initiates a new project by uploading a Statement of Work (SOW) document for an existing client. The system accepts the file, associates it with a new project record, stores the file securely, and triggers the asynchronous AI processing workflow via a message queue.

## 1.4 Type

🔹 BusinessProcess

## 1.5 Purpose

To kickstart the core business workflow of the platform by ingesting a new project's requirements via an SOW, as defined in REQ-FUN-002.

## 1.6 Complexity

Medium

## 1.7 Priority

🚨 Critical

## 1.8 Frequency

OnDemand

## 1.9 Participants

- REPO-FRONTEND-SPA
- REPO-API-MONOLITH
- REPO-DB-POSTGRES
- REPO-MQ-RABBITMQ
- REPO-INFRA-AWS

## 1.10 Key Interactions

- Admin selects a client and uploads a .docx or .pdf file via the UI.
- Frontend sends the file and associated Client ID to the API.
- API creates a new Project record (linked to the Client) and an SOW record in the database with a 'Processing' status.
- API uploads the original SOW document to a secure, private AWS S3 bucket with a unique key.
- API publishes a 'SowUploaded' event to a durable message queue (RabbitMQ), containing the sowDocumentId and projectId.

## 1.11 Triggers

- An Admin uploads an SOW document to start a new project.

## 1.12 Outcomes

- A new Project and SOW record are created and persisted in the database.
- The original SOW file is securely stored in AWS S3.
- An event is published, decoupling the upload from the long-running AI analysis.

## 1.13 Business Rules

- A Project cannot be created without being associated with an existing, active Client (REQ-BUS-001).
- The uploaded file must be of a supported format (.docx, .pdf) and within a size limit.

## 1.14 Error Scenarios

- The file upload fails due to network issues or size limits.
- The message queue is unavailable, causing the entire transaction to be rolled back.
- The S3 upload fails due to permission issues.

## 1.15 Integration Points

- AWS S3 for secure file storage.
- RabbitMQ (or similar message broker) for asynchronous event publishing.

# 2.0 Details

## 2.1 Diagram Id

SEQ-BP-001

## 2.2 Name

Admin Initiates New Project via SOW Upload

## 2.3 Description

A comprehensive sequence detailing the business process of initiating a new project. An authenticated System Administrator uploads a Statement of Work (SOW) document for a pre-existing client. The backend API orchestrates a transactional operation to persist project metadata, store the SOW file in a secure object store, and publish an event to a message broker to trigger an asynchronous AI analysis workflow. This sequence represents the primary entry point for the platform's core value stream.

## 2.6.0.0 Notes

### 2.6.1.0 Content

#### 2.6.1.1 Content

Transactional Boundary: The database operations (steps 4, 5) and the S3 upload (step 6) must be treated as a single atomic unit. A failure in any of these steps must trigger a rollback of the database transaction and deletion of the S3 object if it was created.

#### 2.6.1.2 Position

right

#### 2.6.1.3 Participant Id

REPO-API-MONOLITH

#### 2.6.1.4 Sequence Number

3

### 2.6.2.0 Content

#### 2.6.2.1 Content

Reliable Event Dispatch: Publishing the 'SowUploaded' event (step 8) occurs only after the database transaction is successfully committed. This prevents a scenario where a consumer tries to process an event for data that doesn't exist. Failure to publish is a critical state that must be alerted on, with a recovery process (e.g., a background job that finds unprocessed SOWs) in place.
