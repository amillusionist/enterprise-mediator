# 1 Overview

## 1.1 Diagram Id

SEQ-IF-001

## 1.2 Name

Client Pays Initial Invoice via Stripe

## 1.3 Description

Once a project is awarded, an invoice is generated and sent to the client. The client clicks a secure payment link, is redirected to a Stripe-hosted checkout page, and completes the payment. A webhook from Stripe asynchronously confirms the payment, which updates the project status.

## 1.4 Type

🔹 IntegrationFlow

## 1.5 Purpose

To securely process client payments for projects and hold funds in escrow via Stripe Connect, fulfilling the financial workflow requirements of REQ-FUN-004 and REQ-INT-002.

## 1.6 Complexity

High

## 1.7 Priority

🚨 Critical

## 1.8 Frequency

OnDemand

## 1.9 Participants

- REPO-API-MONOLITH
- REPO-EXT-SES
- REPO-EXT-STRIPE
- REPO-DB-POSTGRES

## 1.10 Key Interactions

- An Admin triggers invoice creation for an 'Awarded' project.
- API generates the invoice details and creates a payment intent with Stripe.
- API sends an email with a secure Stripe payment link to the client.
- Client clicks the link and completes payment on Stripe's secure, hosted page.
- Stripe processes the payment and sends a 'payment_intent.succeeded' webhook to a dedicated, secure endpoint on the API.
- API's webhook handler first verifies the webhook's signature for authenticity.
- The handler checks the event ID to ensure idempotent processing.
- API records the transaction in the internal ledger.
- API updates the project status to 'Active'.
- API logs the transaction and status change in the audit trail.

## 1.11 Triggers

- Stripe sends a webhook upon successful client payment.

## 1.12 Outcomes

- Client funds are securely captured and held in escrow via Stripe Connect.
- The project status automatically moves to 'Active', signaling that work can officially begin.
- A detailed financial transaction record is created for accounting.

## 1.13 Business Rules

- A project status can only be moved to 'Active' after the initial client invoice has been successfully paid (REQ-BUS-001).
- All financial operations must be atomic and idempotent (REQ-FUN-004).
- The webhook handler must verify the signature to prevent fraudulent requests.

## 1.14 Error Scenarios

- The client's payment fails on Stripe's end.
- The Stripe webhook is delayed or fails to arrive, requiring a reconciliation process.
- The webhook handler fails transiently and must be retried by a queuing system (AWS SQS).

## 1.15 Integration Points

- Stripe Connect API and Webhooks.

# 2.0 Details

## 2.1 Diagram Id

SEQ-IF-001

## 2.2 Name

Implementation: Client Initial Invoice Payment via Stripe Webhook

## 2.3 Description

Technical sequence for processing a client's initial invoice payment. An internal admin triggers invoice generation, which creates a Stripe Checkout Session and emails a payment link to the client. The client pays via Stripe's hosted page. Stripe sends a 'payment_intent.succeeded' webhook, which is received by a secure, idempotent handler. The handler verifies the webhook, updates the internal ledger and project status within an atomic database transaction, and logs the event.
