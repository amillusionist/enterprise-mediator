# 1 Overview

## 1.1 Diagram Id

SEQ-EP-001

## 1.2 Name

Asynchronous SOW Processing by Worker

## 1.3 Description

A dedicated background worker service consumes the 'SowUploaded' event. It retrieves the SOW document and orchestrates the complete AI pipeline: PII sanitization via AWS Comprehend, structured data extraction via OpenAI, and vector embedding generation. The results are then persisted to the database.

## 1.4 Type

🔹 EventProcessing

## 1.5 Purpose

To perform the long-running, resource-intensive AI analysis of the SOW (REQ-FUN-002) without blocking the user-facing API, ensuring system responsiveness and scalability.

## 1.6 Complexity

High

## 1.7 Priority

🚨 Critical

## 1.8 Frequency

OnDemand

## 1.9 Participants

- REPO-MQ-RABBITMQ
- REPO-SVC-AIWORKER
- REPO-INFRA-AWS
- REPO-EXT-COMPREHEND
- REPO-EXT-OPENAI
- REPO-DB-POSTGRES

## 1.10 Key Interactions

- Worker consumes a 'SowUploaded' message from the queue.
- Worker downloads the original SOW from S3 using the ID from the message.
- Worker calls AWS Comprehend to identify and sanitize PII, creating a sanitized text version.
- Worker stores the new sanitized SOW version in S3.
- Worker sends the sanitized text to an OpenAI model (e.g., GPT-4) to extract structured data (skills, deliverables, scope).
- Worker uses an OpenAI embedding model to generate vector embeddings for the extracted skills.
- Worker updates the SOW and Project Brief records in the database with the extracted data, embeddings, and sets the status to 'ReviewPending'.

## 1.11 Triggers

- A 'SowUploaded' event is available in the message processing queue.

## 1.12 Outcomes

- The SOW record is updated with extracted structured data and a 'ReviewPending' status, ready for HITL.
- Vector embeddings for required project skills are stored in the database.
- A sanitized version of the SOW is securely stored in S3.
- The Admin receives a notification that SOW processing is complete and ready for review.

## 1.13 Business Rules

- Sanitization must consistently replace PII with placeholders like [PERSON_NAME] as per the defined policy (REQ-FUN-002).
- Both original and sanitized SOWs must be stored securely with different access policies.
- The operation must be idempotent to handle potential message redelivery.

## 1.14 Error Scenarios

- External AI APIs are unavailable or return errors, triggering a retry policy.
- The document format is unparsable, causing a permanent failure.
- After all retries fail, the message is moved to a Dead-Letter Queue (DLQ) for manual intervention.

## 1.15 Integration Points

- AWS S3
- AWS Comprehend API
- OpenAI API

## 2.3 Description

Detailed sequence for the event-driven AI worker service (REPO-SVC-AIWORKER) consuming a 'SowUploaded' event. The worker orchestrates a multi-step pipeline involving idempotent checks, document retrieval from AWS S3, PII sanitization via AWS Comprehend, structured data extraction and vector embedding generation via OpenAI, and finally, persisting the results to the PostgreSQL database in a single transaction before acknowledging the message.
