# 1 Overview

## 1.1 Diagram Id

SEQ-CF-001

## 1.2 Name

Project Status Transition and Auditing

## 1.3 Description

Throughout a project's lifecycle, any change to its status (e.g., from 'Active' to 'In Review') is validated against business rules and recorded. A corresponding immutable entry is created in a dedicated audit log to provide a complete, traceable history of the project.

## 1.4 Type

🔹 ComplianceFlow

## 1.5 Purpose

To enforce the defined project lifecycle (REQ-FUN-003) and maintain a comprehensive, immutable audit trail for all critical actions, supporting security, dispute resolution, and compliance needs (REQ-FUN-005, REQ-BUS-003).

## 1.6 Complexity

Medium

## 1.7 Priority

🔴 High

## 1.8 Frequency

OnDemand

## 1.9 Participants

- REPO-API-MONOLITH
- REPO-DB-POSTGRES
- emp_observability_db

## 1.10 Key Interactions

- A user action (e.g., Admin approving a deliverable) triggers a status change request.
- The API's application service receives the request.
- The service's business logic validates if the requested transition is valid based on the current state and user role.
- The service updates the Project entity's status in the primary PostgreSQL database.
- Concurrently, the service captures the 'before' and 'after' state of the project entity.
- An Audit Log entry is created containing the responsible user ID, timestamp, IP address, action type, target entity, and the before/after state snapshot.
- The Audit Log entry is written to the dedicated observability database (OpenSearch).

## 1.11 Triggers

- Any system event or user action that changes a project's lifecycle status.

## 1.12 Outcomes

- The project's status is updated in the primary operational database.
- A detailed, immutable audit log of the change is created and stored in a separate, secure system.

## 1.13 Business Rules

- Project state transitions must follow the predefined, valid lifecycle path.
- No user, including System Administrator, can modify or delete audit log entries (REQ-SEC-001).
- The audit log write must succeed for the entire operation to be considered successful.

## 1.14 Error Scenarios

- An invalid state transition is attempted and rejected by the API.
- The audit log database is unavailable, causing the entire transaction to be rolled back to maintain consistency.

## 1.15 Integration Points

- Primary PostgreSQL DB and the separate Observability (OpenSearch) DB.

## 2.6.0.0 Notes

### 2.6.1.0 Content

#### 2.6.1.1 Content

Compliance as Code: The business logic in Step 3 is a direct implementation of 'Compliance as Code'. The project state machine is codified and automatically enforced, ensuring all transitions adhere to the defined lifecycle (REQ-FUN-003).

#### 2.6.1.2 Position

### 2.6.2.0 Content

#### 2.6.2.1 Content

Data Immutability: The OpenSearch index for audit logs must have a policy that denies update and delete operations for all roles except a break-glass administrative role, enforcing the immutability requirement from REQ-SEC-001.

#### 2.6.2.2 Position

bottom-right

#### 2.6.2.3 Participant Id

emp_observability_db

#### 2.6.2.4 Sequence Number

7

### 2.6.3.0 Content

#### 2.6.3.1 Content

Transactional Consistency: The business rule 'audit log write must succeed for the entire operation to be successful' presents a distributed transaction challenge. The chosen pattern prioritizes availability of the primary system. It commits the primary state change first, then handles audit failures with a guaranteed retry mechanism (e.g., a Dead Letter Queue and worker). This ensures the operational state is never blocked by an audit system outage, while a separate reconciliation process guarantees eventual consistency of the audit trail.
