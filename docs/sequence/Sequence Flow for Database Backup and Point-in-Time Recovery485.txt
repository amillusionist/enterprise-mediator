# 1 Overview

## 1.1 Diagram Id

SEQ-RF-001

## 1.2 Name

Database Backup and Point-in-Time Recovery

## 1.3 Description

The system's primary AWS RDS PostgreSQL database is automatically backed up daily via snapshots. In case of a disaster or data corruption event, an operator can restore the database to a specific point in time from these backups to a new instance.

## 1.4 Type

🔹 RecoveryFlow

## 1.5 Purpose

To ensure business continuity and data durability by meeting the Recovery Time Objective (RTO) of 4 hours and Recovery Point Objective (RPO) of 15 minutes, as defined in REQ-NFR-002.

## 1.6 Complexity

High

## 1.7 Priority

🚨 Critical

## 1.8 Frequency

Daily

## 1.9 Participants

- REPO-INFRA-AWS
- REPO-DB-POSTGRES

## 1.10 Key Interactions

- (Backup) AWS RDS automatically creates a daily snapshot of the database and retains it for 35 days.
- (Backup) Transaction logs are continuously backed up to S3, enabling point-in-time recovery (PITR) for the last 14 days.
- (Recovery) An operator declares a disaster and initiates the recovery process via the AWS console or CLI.
- Operator selects the target snapshot and the specific point in time (e.g., timestamp just before the incident).
- AWS RDS provisions a new database instance and restores the data from the chosen snapshot and relevant transaction logs.
- Once the new DB is available, infrastructure configuration (e.g., Terraform variables) is updated to point all application services to the new database endpoint.
- The application is redeployed or restarted to use the new connection string.

## 1.11 Triggers

- A scheduled backup policy (for backups).
- A declared disaster or critical data loss event (for recovery).

## 1.12 Outcomes

- System data is restored to a consistent state from a point before the incident, meeting the RPO.
- The application is brought back online and functional, meeting the RTO.

## 1.13 Business Rules

- Daily snapshots must be retained for 35 days (REQ-NFR-002).
- Point-in-time recovery must be enabled for a retention period of at least 14 days (REQ-NFR-002).
- The disaster recovery plan must be documented and tested quarterly to verify the RTO and RPO.

## 1.14 Error Scenarios

- The backup process fails silently, which must be monitored and alerted on.
- The restored data is discovered to be inconsistent or incomplete after recovery, requiring a root cause analysis.

## 1.15 Integration Points

- AWS RDS Backup and Recovery services.

## 2.1 Diagram Id

SEQ-US-095

## 2.2 Name

Client Milestone Approval via Secure Link

## 2.3 Description

This sequence diagram details the end-to-end technical flow for a Client Contact approving a project milestone using a secure, single-use, time-limited link delivered via email. The process is designed to be low-friction, operating without requiring the client to log in. The sequence covers token validation, atomic state transition of the milestone, token invalidation, audit logging, and final user notification.

#### 2.6.3.1.0.0 Content



```
**Idempotency**
The API endpoint for approval (Step 6) must be idempotent. The second token validation within the service logic (Step 7.1) and the atomic transaction ensure that if two identical requests arrive simultaneously, only the first one will succeed, and the second will fail with a token-used error (e.g., 409 Conflict), preventing duplicate state changes.
```
